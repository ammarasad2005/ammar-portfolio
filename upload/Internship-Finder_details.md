## Overview

Internship Finder is an asynchronous, "deterministic-first" research agent that discovers software-engineering internships on the web, normalizes and deduplicates them, scores them against a student's profile, and emails a digest of the best matches. The product philosophy (`.rules`, `project-context.md`) is unambiguous: it targets **10–50 Pakistani university students**, prioritizes discovery quality over scale, and treats AI (Gemini) as an expensive, scarce resource that must be cached, batched, and only invoked after deterministic SQL/regex/normalization has done the heavy lifting.

The user-facing surface is a small Next.js App Router site: landing page (`/`), Supabase email/password auth (`/auth/login`, `/auth/signup`), a multi-path onboarding flow (`/onboarding` — Resume upload, LinkedIn URL, Quick Start, plus a Conversational fallback), a dashboard (`/dashboard`) showing profile confidence score + session history, a live session timeline (`/dashboard/sessions/[id]`) powered by Supabase Realtime, and a matches page (`/dashboard/sessions/[id]/matches`) with AI-generated explanations and Save/Apply/Reject feedback buttons. The heavy lifting — Google CSE search, HTML extraction, canonicalization, persistence, AI matching, and SMTP email — never runs inside Vercel; it runs in a GitHub Actions worker triggered by a `repository_dispatch` from the Next.js API route `/api/sessions/trigger`. When the worker finishes, a Supabase database webhook fires `/api/webhooks/session-completed`, which sends the email digest. The whole architecture is engineered around a **$0 operational budget** (free Vercel, free Supabase tier, 100 free Google CSE queries/day, free GitHub Actions minutes, Gmail SMTP).

## Architecture

The codebase is organized by **feature modules** under `src/features/` (`agents.md` mandates this), each with `services/`, `components/`, `types/`, and sometimes `actions/` or `schemas/` subfolders. Cross-cutting Supabase clients live in `src/lib/supabase/` (`client.ts` browser, `server.ts` SSR cookies, `types.ts` hand-written `Database` interface). The layers and their interactions:

- **UI layer (`src/app/`):** Server Components for dashboard/session/matches pages, Client Components for auth forms, onboarding wizard, session timeline (Subabase Realtime subscriptions in `SessionDetail.tsx`), and the `StartSessionButton`. Styled with co-located `*.module.css`.
- **Auth/middleware (`src/middleware.ts`):** Supabase SSR `createServerClient` refreshes the session on every request; redirects unauthenticated users away from `/dashboard` and `/onboarding`, and authenticated users away from `/auth`.
- **API routes:** `POST /api/sessions/trigger` (auth-gated, inserts a `search_sessions` row at status `pending`, then `fetch`-es the GitHub `repository_dispatch` API) and `POST /api/webhooks/session-completed` (verifies `x-webhook-secret`, filters for `UPDATE` on `search_sessions` with `status='completed'`, delegates to `NotificationService`).
- **Brain (`src/features/brain/`):** `GeminiService` (singleton, `gemini-2.5-flash`, Zod-validated structured output, retry with exponential backoff), `AiCacheService` (SHA-256 cache keys in the `ai_cache` table), `DomainExpansionService`, `QueryGenerationService`, `QueryRankingService` (deterministic — still effectively a mock), `ResearchPlanBuilder` (orchestrator), and `FeedbackProfileBuilder` (aggregates a user's past saved/rejected matches into positive/negative company/tag/keyword sets).
- **Discovery (`src/features/discovery/`):** `ProviderRegistry` (in-memory map with circuit breaker + daily quota reset), `ProviderRouter` (weighted random selection + failover), `SearchWavePlanner`, `QueryBudgetManager`, `QueryDeduplicationEngine`, `SearchCoverageAnalyzer` (round-robin category diversity), `SearchCostEstimator`, `ProviderSelectionEngine`.
- **Worker (`src/features/worker/`):** `WorkerLifecycle.runSession()` is the orchestrator; `ResearchExecutor` runs the wave with retry/backoff/failover; `CostTracker` accumulates token cost; `providers/` holds the `SearchProvider` interface, `GoogleCSEProvider` (real), and `MockSearchProvider` (10% random failure for chaos testing).
- **Extraction / canonicalization / persistence (`src/features/extraction|canonicalization|persistence/`):** `ExtractionEngine` → `PageFetcher` → `PageClassifier` → `InternshipExtractor` (cheerio + JSON-LD) → `ExtractionConfidenceScorer` → `CandidateValidator`; `CanonicalKeyGenerator` (tri-factor `company:role:location` hash) + `InternshipNormalizer` + `DeduplicationEngine` + `InternshipMergeStrategy` + `SourceAggregator`; `InternshipPersistenceService` → `InternshipRepository` → `UpsertStrategy` + `InternshipSourcePersistenceService` + `DatabaseMappingLayer` (deterministic tag generation).
- **Matching (`src/features/matching/`):** `MatchEngine.executeDeltaBatch()` (internship-centric delta batch), `MatchScorer` (85-pt deterministic heuristic + 15-pt AI boost), `MatchExplanationService` (Gemini, top-5 only), `MatchRepository` (upsert preserving `user_feedback`), `MatchService` (server-side read for UI), `feedback.action.ts` (Next.js Server Action).
- **Notifications (`src/features/notifications/`):** `NotificationService` (threshold filter + idempotency + Nodemailer) and `NotificationEmailTemplate` (inline-styled HTML string).
- **Onboarding / research / auth / dashboard features** hold the React flows and `ProfileService` / `SessionService`.

Two Supabase clients are used: the SSR cookie client (`@supabase/ssr`, anon key, RLS-enforced) for the UI, and a service-role client (`@supabase/supabase-js`, `SUPABASE_SERVICE_ROLE_KEY`, RLS-bypassing) created in `run-worker.ts` and `NotificationService` for privileged worker writes.

## Application Model

- **Routing:** Next.js App Router with `src/app/`. Server Components by default; Client Components opted in via `'use client'` for auth forms, onboarding wizard, session timeline, start-session button, and feedback buttons.
- **Rendering mix:** Pages are server-rendered (SSR) and revalidated via `revalidatePath` after feedback (`feedback.action.ts`). The session timeline is **CSR with Supabase Realtime WebSockets** — `SessionDetail.tsx` subscribes to `postgres_changes` on `research_session_events` (INSERT) and `search_sessions` (UPDATE) and appends events live. No SSG/ISR is configured (`next.config.ts` is empty). The dashboard and matches pages read directly from Supabase in Server Components.
- **Auth:** Supabase Auth (email/password), session cookies managed by `@supabase/ssr` and refreshed in `src/middleware.ts`. RLS is enforced on every table (see `supabase/migrations/20260618000000_initial_schema.sql`).
- **Execution model split (the defining architectural choice):** Vercel serverless handles only short UI/API work (auth, onboarding persistence, session creation, GitHub dispatch, webhook receipt, match rendering). **All long-running research, scraping, AI calls, and matching run asynchronously in GitHub Actions** via `repository_dispatch` → `npx tsx src/scripts/run-worker.ts`. This explicitly sidesteps Vercel's 5-minute function timeout (GHA allows up to 6 hours). `CURRENT_STATE.md` and `project-context.md` both state this is mandatory.
- **Server-only isolation:** `WorkerLifecycle`, `MatchEngine`, `MatchRepository`, `MatchExplanationService`, and `FeedbackProfileBuilder` import `'server-only'`. To let `tsx` execute them as a standalone Node script, `tsconfig.json` maps `"server-only"` to `./src/mocks/server-only-mock.ts` (an empty file), a clever compile-time-vs-runtime trick.
- **Background trigger pattern:** UI → `POST /api/sessions/trigger` → insert `search_sessions` (pending) → GitHub `repository_dispatch` (`trigger-worker`) → GHA clones repo, runs worker → worker updates session status through the lifecycle → Supabase DB webhook → `/api/webhooks/session-completed` → email.

## Data Model

Eleven PostgreSQL tables defined in `supabase/migrations/20260618000000_initial_schema.sql` (Postgres 17), with three follow-up migrations adding notification columns, idempotency columns, API grants, and performance indexes. Five enums: `search_session_status` (`pending|running|completed|failed|cancelled`), `remote_preference_type`, `source_type_enum`, `cache_type_enum`, `user_feedback_type` (`applied|rejected|saved`). The TypeScript mirror lives in `src/lib/supabase/types.ts` (manually maintained — `TRANSITION_SNAPSHOT.md` flags this as drift risk).

- **`profiles`** — PK `id` (FK to `auth.users(id) ON DELETE CASCADE`), `first_name`, `last_name`, `location_preference`, `remote_preference`, `duration_preference`, `paid_preference`, `confidence_score INTEGER CHECK 0–100` (gates whether research can begin), `email_notifications_enabled BOOLEAN DEFAULT TRUE`, `notification_threshold INTEGER DEFAULT 75 CHECK 0–100` (added in `20260620000000_profile_notification_settings.sql`). `updated_at` trigger.
- **`profile_skills`** — `profile_id`, `skill_name`, `source` (`resume_parse|manual|ai_enrichment|linkedin` per the TS type). 1:N to profiles.
- **`profile_projects`** — `project_name`, `description`, `technologies TEXT[]`, `project_url`, `source`. Strongest signal for semantic matching.
- **`search_sessions`** — `status`, `started_at`, `completed_at` (CHECK `completed_at IS NULL OR completed_at >= started_at`), `notification_sent BOOLEAN DEFAULT FALSE`, `notification_sent_at TIMESTAMPTZ` (added in `20260620100000_session_notification_idempotency.sql`). Acts as the orchestrator record for the worker.
- **`research_session_events`** — `event_type` TEXT, `message`, `metadata JSONB`. Powers the live UI timeline.
- **`research_runs`** — `query_used`, `search_provider`, `depth_level INTEGER CHECK ≥0`, `results_found INTEGER CHECK ≥0`. Telemetry per search execution.
- **`internships`** — `canonical_key TEXT UNIQUE` (format `normalized_company:normalized_role:normalized_location`), `company_name`, `role_title`, `location`, `description`, `application_url`, `tags TEXT[]` (GIN index for fast SQL pre-filtering), `is_active BOOLEAN`. Indexed on `is_active`, `tags` (GIN), and `discovered_at` (B-Tree, added `20260619000000`).
- **`internship_sources`** — provenance; `source_url`, `source_type` enum, `is_primary BOOLEAN` (true for official career pages), `UNIQUE(internship_id, source_url)`.
- **`matches`** — `profile_id`, `internship_id`, `session_id`, `semantic_score INTEGER CHECK 0–100`, `explanation TEXT` (Gemini "Why this fits you"), `user_feedback` enum (nullable), `UNIQUE(profile_id, internship_id, session_id)`. Composite index on `(session_id, profile_id)` and descending index on `semantic_score` for the matches page.
- **`ai_cache`** — `cache_key TEXT UNIQUE` (SHA-256 hash), `cache_type` enum, `output_value JSONB`, `expires_at`. **No RLS policies** — service-role only.
- **`research_usage_logs`** — `profile_id` (nullable for guest tracking), `session_id`, `tokens_used`. Time-series quota/limits log.

**RLS strategy** (from `initial_schema.sql` and `20260621000001_api_grants.sql`): strict user-ownership (`auth.uid() = profile_id`) for profiles/skills/projects/sessions/matches/usage-logs; global read-only for `internships`/`internship_sources`; session-scoped read for `research_session_events`/`research_runs`; `ai_cache` has no policies (default-deny for anon/authenticated, service-role only). The service role bypasses RLS entirely for worker writes.

## Pipelines / Flows / Execution Path

**1. Onboarding (`src/features/onboarding/`):** `OnboardingFlow.tsx` is a state machine: `method_selection` → `data_collection` (Resume/LinkedIn/QuickStart) → either `review` (if confidence ≥ 70) or `conversational_enrichment` (if < 70) → `review`. `calculateConfidenceScore` (`utils/confidenceScore.ts`) sums basic info (20) + preferences (10) + skills (5 each, cap 30) + projects (15 each, cap 40). *Inferred:* the `ResumeUpload`, `LinkedInInput`, and `ConversationalUI` components are currently mocks — they `setTimeout` and return hardcoded data ("Ammar", React, Next.js, Python, JavaScript) rather than actually parsing a PDF or fetching LinkedIn. Only `QuickStartForm` captures a real domain string. `ProfileReview` (not read in detail) persists via `ProfileService.saveOnboardingProfile`, which upserts the profile then delete-and-reinserts skills and projects.

**2. Session trigger (`src/features/research/components/StartSessionButton.tsx` → `src/app/api/sessions/trigger/route.ts`):** Client `fetch` POSTs `{profileId}` to the API route. The route verifies `auth.getUser()`, checks `profileId === user.id`, inserts a `search_sessions` row at `pending`, publishes a `session_created` event, then POSTs to `https://api.github.com/repos/{owner}/{repo}/dispatches` with `event_type: 'trigger-worker'` and `client_payload: { session_id, profile_id }` using `GITHUB_PAT`. UI redirects to `/dashboard/sessions/{sessionId}`.

**3. Worker execution (`.github/workflows/worker.yml` → `src/scripts/run-worker.ts` → `WorkerLifecycle.runSession`):** GHA (Node 22, `npm ci`) runs `npx tsx src/scripts/run-worker.ts --sessionId --profileId`. The script builds a service-role Supabase client and calls `WorkerLifecycle.runSession(sessionId, profileId, supabase)`. Inside `runSession`:
   - Mark session `running`, publish `worker_assigned`.
   - `FeedbackProfileBuilder.build` — aggregates the user's past `matches` (joined with `internships`) into `positiveCompanies`, `positiveTags`, `negativeCompanies`, `negativeKeywords` Sets.
   - `ResearchPlanBuilder.buildPlan` — `ProfileService.getProfile` → `DomainExpansionService.expandProfile` (Gemini, cached, deterministic fallback) → `QueryGenerationService.generateQueries` (Gemini with feedback-prompt injected, cached, deterministic fallback) → `QueryRankingService.rankQueries` → cap to top 15 queries.
   - `SearchWavePlanner.buildWave(1, queries, new Set(), {maxQueriesPerWave: 15, maxCostPerWave: 100})` — dedup → coverage round-robin → `ProviderRouter.selectProvider` (weighted) → budget cap. `CostTracker.logTokens(wave.totalEstimatedCost)`.
   - `ResearchExecutor.executeWave` — for each task: `provider.search(query.text)`, 500ms rate-limit sleep, `ProviderRegistry.reportSuccess`, `SessionService.createRun(...)` (writes a `research_runs` row). On failure: 3 retries with exponential backoff for transient errors; for fatal 429/403/400, immediately failover via `ProviderRouter.getFailover`.
   - `CostTracker.flushToDatabase` — inserts a `research_usage_logs` row.
   - `MatchEngine.executeDeltaBatch(lastRunTime = now - 24h)` — fetches `internships WHERE discovered_at > lastRunTime AND is_active`; fetches active profiles via `search_sessions WHERE status IN ('running','pending')` joined to `profiles`/`profile_skills`/`profile_projects`; for each profile×internship pair, `MatchScorer.score` (location 20 + tags 20 + title 20 + projects 25 + feedback ±[−50,+20]); keep matches with `total_score ≥ 25`; sort; top 5 per profile get `MatchExplanationService.generateExplanation` (Gemini `score_boost` 0–15 + explanation); `MatchRepository.upsertBatch` (omits `user_feedback` to preserve user actions).
   - Mark session `completed`.

**4. Notification (`/api/webhooks/session-completed` → `NotificationService`):** Supabase DB webhook fires on `search_sessions` UPDATE. The route verifies `x-webhook-secret`, checks `type==='UPDATE' && table==='search_sessions' && record.status==='completed'`, and calls `NotificationService.sendSessionSummaryEmail`. The service short-circuits if `notification_sent` is already true (idempotency), checks `email_notifications_enabled`, fetches matches with `semantic_score >= notification_threshold` (default 75), renders the inline HTML template, fetches the user's email via `supabase.auth.admin.getUserById`, sends via Nodemailer, then sets `notification_sent=true` + `notification_sent_at`.

**5. Match consumption & feedback (`/dashboard/sessions/[id]/matches` → `MatchCard` → `FeedbackButtons`):** `MatchService.getSessionMatches` (SSR, joined query) renders `MatchCard`s with `ScoreBar`, `TagList`, AI explanation, and `FeedbackButtons` (Apply / Save / Reject). `FeedbackButtons` uses `useTransition` for optimistic updates and calls the `recordMatchFeedback` Server Action, which updates `matches.user_feedback` filtered by `id AND profile_id = auth.uid()` (belt-and-suspenders over RLS) and `revalidatePath`s the page. This feedback is what `FeedbackProfileBuilder` consumes in the next session — closing the loop.

## Problem Solved

The repo targets a concrete, narrow pain point: Pakistani university students (the README/`project-context.md` say 10–50 of them) struggle to surface high-quality software-engineering internships because relevant postings are scattered across official company career pages, LinkedIn, Rozee.pk, Mustakbil, Indeed, and Glassdoor, with heavy duplication and no personalization. Generic job-board search is keyword-only and surfaces mostly senior roles or noise. Internship Finder solves this by (a) auto-expanding a sparse student profile into a richer set of search intents via Gemini, (b) querying Google CSE (which indexes all of the above) with those intents, (c) deterministically classifying, extracting, validating, and deduplicating the resulting pages, (d) scoring the deduplicated canonical internships against the user's skills/projects/preferences with a transparent 85-point heuristic, (e) selectively using Gemini to add a 15-point semantic boost + a human-readable "Why this fits you" explanation for only the top 5 matches per user, and (f) emailing a digest and letting the user save/reject/apply — which feeds back into the next session's queries and scoring. The architecture also solves the secondary problem of **cost**: a fully personalized, AI-augmented internship agent that costs $0/month to operate at 50-user scale, by exploiting free tiers of Vercel, Supabase, Google CSE (100 queries/day), GitHub Actions, and Gmail SMTP, with aggressive AI caching to keep Gemini calls near zero.

## Novel Aspects

Several aspects are uncommon for a personal/student project:

- **Decoupled UI/worker via GitHub `repository_dispatch`:** instead of a typical cron worker, queue, or serverless function, the Next.js API route literally POSTs to the GitHub REST API to trigger a workflow run, then the GHA runner clones the repo and runs a `tsx` script. This abuses GHA as a free, long-running compute platform ($0, 6-hour limit) — a creative hack to escape Vercel's 5-minute function ceiling without paying for a queue/worker.
- **`server-only` mock path trick:** `tsconfig.json` maps `server-only` to an empty local mock file so the same code that compiles for Next.js can also be executed standalone by `tsx` in GHA, while still preventing accidental import of privileged worker code into the client bundle.
- **Internship-Centric Delta Batch matching (`MatchEngine`):** rather than the naive N×M×Q pattern, it issues exactly two global SQL queries (new internships since last run + active profiles with joined skills/projects), then scores every pair in Node.js memory, calling Gemini only for the top 5 per user. This avoids N+1 connection exhaustion and minimizes AI cost.
- **Feedback-closed loop across the whole system:** user Save/Apply/Reject flows back through `FeedbackProfileBuilder` into both the heuristic scorer (±[−50,+20] adjustment) **and** the upstream Gemini `QueryGenerationService` prompt, so rejection of a company literally biases the next session's search queries away from that company. This turns the app into a personalized recommender rather than a one-shot search.
- **Tri-factor canonical key with collision escape hatches:** `CanonicalKeyGenerator` produces `company:role:location`, but appends a hashed-URL suffix when the company is "Unknown" or the title is generic ("Intern"/"Trainee"), preventing catastrophic over-merging of distinct listings that happen to share a vague label.
- **AI cache as a first-class DB table** (`ai_cache` with SHA-256 keys, TTL, and cache-type enums), default-denied to all roles except service-role — a deliberate quota-protection mechanism, not an afterthought.

## Optimizations / Efficiencies

- **AI caching everywhere:** `DomainExpansionService` and `QueryGenerationService` both hash their sorted, JSON-stringified inputs into `ai_cache` keys and skip the Gemini call on cache hit (`AiCacheService.generateHash` uses SHA-256; default TTL 30 days). Because skills/projects are sorted before hashing, logically equivalent inputs hit the same key.
- **Deterministic fallbacks for every Gemini call:** both `DomainExpansionService.fallbackDeterministic` and `QueryGenerationService.fallbackDeterministic` produce reasonable output (mapped expansions/queries) if Gemini throws or the API key is missing — the system degrades gracefully rather than crashing.
- **Provider circuit breaker + daily quota reset:** `ProviderRegistry` tracks `consecutiveFailures` (3 = `dead`, half-open recovery ping after 5 min) and `usedToday` vs `dailyLimit` with automatic midnight reset — protecting the 100-query/day Google CSE budget.
- **Round-robin category diversity (`SearchCoverageAnalyzer.optimizeCoverage`):** prevents the budget from being consumed entirely by one query category (e.g. all `role_based`) before sampling `technology_based`/`skill_based`/`company_based`.
- **Failover over infinite retry:** fatal 429/403/400 errors skip the backoff loop and immediately reroute to a healthy failover provider (`ProviderRouter.getFailover`) rather than burning the wave on a dead API.
- **Two-query delta batch + in-memory scoring** (described above) minimizes DB round-trips.
- **Targeted indexes:** `idx_internships_discovered_at` (B-Tree) for the delta fetch, `idx_matches_session_profile` (composite) and `idx_matches_semantic_score` (descending) for the matches page's exact query pattern, `idx_internships_tags` (GIN) for fast tag pre-filtering.
- **Idempotent notifications:** the `notification_sent`/`notification_sent_at` columns plus the early-return check in `NotificationService` prevent duplicate emails if Supabase retries the webhook.
- **`user_feedback` preservation on match re-upsert:** `MatchRepository.upsertBatch` deliberately omits `user_feedback` from the payload so regenerating a match never clobbers a user's explicit Save/Apply/Reject.
- **Cheerio + JSON-LD extraction:** `InternshipExtractor` first tries `script[type="application/ld+json"]` with `@type === 'JobPosting'` (structured data) before falling back to `h1`/`title`/`og:site_name` heuristics — far more reliable than pure DOM scraping.

## Design Decisions

- **No Tailwind, CSS Modules only** (`.rules`, `agents.md`, `SESSION_HANDOFF.md`): an explicit aesthetic mandate — "The app must look premium. We are explicitly banned from using TailwindCSS."
- **Deterministic-first AI philosophy** (`.rules`, `project-context.md`, `EXECUTIVE_SUMMARY.md`): "Never use AI when SQL or a dictionary works." AI is restricted to profile analysis, domain expansion, semantic matching, and explanation generation; everything else (filtering, deduplication, validation, scoring) is deterministic.
- **Feature-based directory structure** (`src/features/{auth,onboarding,brain,worker,discovery,extraction,canonicalization,persistence,matching,notifications,research,dashboard}/`) mandated by `agents.md` — each feature owns its services, components, and types.
- **Two Supabase clients with strict key separation:** SSR cookie client (anon key, RLS-enforced) for the UI; service-role client (RLS-bypassing) only instantiated in `run-worker.ts` and `NotificationService`. `PHASE_18_ARCHITECTURE.md` §6 codifies the secret boundary: Vercel gets anon key + GitHub PAT; GHA gets service-role + AI keys; Supabase gets the webhook secret.
- **`search_sessions` as the orchestrator record:** rather than a separate queue table, the session row's `status` enum is the worker's state machine and the webhook trigger.
- **Canonical key = `company:role:location`** (after normalization), with source URLs aggregated into `internship_sources` for provenance — never discarded. Official career pages are preferred over job boards during merge (`InternshipMergeStrategy`).
- **Research plan capped at 15 queries** (`ResearchPlanBuilder`: `queries.slice(0, 15)`) and the wave capped at 15 queries / 100 cost units — a deliberate throttle to fit within Google CSE's 100/day free tier.
- **Gemini `temperature: 0.2`** (`gemini.service.ts`) and `responseMimeType: 'application/json'` with Zod schema validation on every response — chosen for determinism and safety.
- **Top-5-only AI explanations:** `MatchEngine` only calls `MatchExplanationService` for the top 5 matches per user, capping Gemini explanation calls at 5 × (active users) per batch.
- **Postgres 17 + Supabase Realtime + Inbucket** for local dev mail catching (`supabase/config.toml`).

## Notable Implementation Details

- **Gemini SDK usage:** `gemini.service.ts` calls `this.ai.models.generateContent({ model: 'gemini-2.5-flash', contents: prompt, config: { systemInstruction, responseMimeType: 'application/json', temperature: 0.2 } })`, then reads `response.text` (**as a getter, not a function** — `EXECUTIVE_SUMMARY.md` documents that this was a Phase 13 bug where `response.text()` was called as a function). The text is `JSON.parse`d and `schema.parse`d with Zod; on failure it retries up to 2 times with exponential backoff.
- **`FeedbackProfileBuilder`** joins `matches` to `internships` via Supabase's nested select syntax (`select('user_feedback, internships ( company_name, role_title, tags )')`), then defensively handles the fact that PostgREST may return either an object or an array for the joined row (`Array.isArray(internship) ? internship[0] : internship`). Saved/applied → positive companies + tags; rejected → negative companies + title tokens (length > 2) as negative keywords.
- **`MatchScorer` scoring breakdown** (max 100): `scoreLocationAndRemote` (max 20: 10 for remote-pref match + 10 for location match), `scoreTagsAndSkills` (max 20: overlap/3 × 20), `scoreTitleRelevance` (max 20: +10 per matching skill/tech keyword in title, +5 generic software/developer/engineer boost), `scoreProjectRelevance` (max 25: +5 per matching tech in description, +5 per matching project name), `scoreFeedback` (±[−50, +20]: +15 positive company, +min(tag×5,10) positive tags, −50 negative company, −30 negative keyword). `ai_boost` (0–15) is added afterward by `MatchExplanationService` and capped at 100.
- **`InternshipExtractor`** uses cheerio to (1) extract title from `h1`/`title` with regex cleanup of `Careers | …` / `Job Application for …`, (2) prefer JSON-LD `JobPosting` structured data for company + title, (3) fall back to `og:site_name` meta or hostname, (4) extract location from `.location`/`.job-location`/`[data-qa="job-location"]`, (5) find the apply link via `a:contains("Apply"), a:contains("Submit")` and resolve relative URLs, (6) extract description from `.job-description`/`.description`/`#content`/`main` or fall back to all `<p>` text, truncated at 1000 chars.
- **`PageClassifier`** recognizes Pakistani job boards (`rozee.pk`, `mustakbil.com`) alongside global ones (LinkedIn, Indeed, Glassdoor) and ATS platforms (`lever.co`, `greenhouse.io`, `workable.com`), distinguishing `job_board_listing` (specific `/view/`/`/job/`/`/detail/` URLs) from `irrelevant` search-result pages.
- **`UpsertStrategy.execute`** determines whether a row was created vs updated by comparing `created_at` and `updated_at` timestamps — if they're within 1 second, it was an INSERT (clever workaround for Supabase not returning an insert/update discriminator).
- **`GoogleCSEProvider.search`** writes every raw Google CSE JSON response to `.debug/search_responses/google_cse_<timestamp>_<query>.json` on disk for debugging — a useful forensic trail when extraction fails on real DOMs.
- **`NotificationService`** uses `supabase.auth.admin.getUserById(profileId)` to fetch the user's email (since `profiles` doesn't store email — it lives in `auth.users`), which only works because the service uses the service-role key.
- **CRITICAL — Inferred from grep inspection:** the extraction → canonicalization → persistence pipeline appears to be **fully built but never wired into the worker**. `ResearchExecutor.executeTaskWithRetry` calls `provider.search(task.query.text)` and increments `metrics.resultsFound`, then writes a `research_runs` row, but **never passes the `results` to `ExtractionEngine.processSearchResult`**, never instantiates `DeduplicationEngine`, and never calls `InternshipPersistenceService.persistBatch`. A grep for `processSearchResult|persistBatch|new DeduplicationEngine|new InternshipPersistenceService|new ExtractionEngine` across `src/` returns only the definitions, no callers. Consequently `MatchEngine.executeDeltaBatch` (which queries `internships WHERE discovered_at > lastRunTime`) would find zero rows on every run, the `matches` table would stay empty, and no notification email would ever qualify. The docs (`EXECUTIVE_SUMMARY.md`, `CURRENT_STATE.md`, `PHASE_SUMMARY.md`, `TRANSITION_SNAPSHOT.md`) claim "Everything works" and "the full pipeline is" operational, but the code as cloned shows a gap between the documented state and the actual wiring. `PHASE_18_ARCHITECTURE.md` §8 itself flags "Risk: Google CSE returns empty or heavily obfuscated job board DOMs, breaking the deterministic extraction logic" — implying extraction was expected to run during the first real run, which aligns with this being an unwired seam rather than a removed feature. (This is an inference from static code reading; the running behavior may differ if a caller exists outside the committed tree.)

## Tradeoffs and Limitations

- **Extraction/persistence not invoked by the worker** (see Notable Implementation Details): the documented end-to-end pipeline is not actually connected in the current code. Until `ResearchExecutor` is extended to feed results through `ExtractionEngine → DeduplicationEngine → InternshipPersistenceService`, the `internships` and `matches` tables will not be populated by a real session. This is the single biggest gap and is the explicit target of the planned "Phase 18 Production Bootstrap & First Real Run."
- **No scheduler/cron:** there is no `schedule:` trigger in `.github/workflows/worker.yml` — the worker runs **only** on `repository_dispatch` from the UI. There is no periodic re-crawl, no overnight refresh, no automatic re-matching of newly discovered internships against existing profiles. A user must click "Start New Discovery" each time.
- **Single real search provider:** only `GoogleCSEProvider` is wired to a live API; `MockSearchProvider` is reused for both `google_mock` and `linkedin_mock` registry entries. The "pluggable multi-provider" architecture is in place but only one real provider exists, capped at 100 free queries/day. `QueryRankingService` is still effectively a mock (deterministic category-based scoring; `TRANSITION_SNAPSHOT.md` confirms it "returns priority score 0 for everything" before the deterministic heuristic — though the code now adds category-based points, the doc still lists it as mocked).
- **Onboarding is mock UI:** `ResumeUpload`, `LinkedInInput`, and `ConversationalUI` return hardcoded data after a `setTimeout` — no PDF parsing, no LinkedIn scraping, no actual Gemini conversational enrichment. Only `QuickStartForm` captures real input. The `Profile Confidence Score` threshold (70) and the conversational fallback flow exist, but the enrichment itself is faked.
- **Sequential Gemini calls:** `MatchEngine` loops over the top 5 matches per profile sequentially in a `for` loop, and `MatchExplanationService.generateExplanation` is awaited inside it. `CURRENT_STATE.md`, `NEXT_PHASE.md`, and `TRANSITION_SNAPSHOT.md` all flag this as a scaling bottleneck for GHA runner minutes.
- **N+1 feedback aggregation:** `MatchEngine.fetchActiveProfiles` calls `FeedbackProfileBuilder.build` once per active session inside a `for` loop (`await FeedbackProfileBuilder.build(this.supabase, session.profile_id)`), which is an N+1 query pattern. Safe for 50 users, problematic at 5,000 (acknowledged in `NEXT_PHASE.md`).
- **`matches` table has no TTL:** `CURRENT_STATE.md` and `TRANSITION_SNAPSHOT.md` note the absence of a 30-day deletion job — the table will grow unbounded.
- **`tags` overwrite on upsert:** `DatabaseMappingLayer.toInternshipRow` always regenerates `tags` deterministically from title+description, so any AI-enriched tags added later would be destructively overwritten (`CURRENT_STATE.md` flags this).
- **Manual type drift:** `src/lib/supabase/types.ts` is hand-maintained rather than generated by the Supabase CLI, so it can silently diverge from `initial_schema.sql` (`TRANSITION_SNAPSHOT.md`).
- **Postgres dead-tuple bloat:** the heavy `ON CONFLICT DO UPDATE` upserts on `internships` will cause bloat without `VACUUM` tuning (`PHASE_SUMMARY.md`).
- **UI is largely unstyled:** bare CSS Modules, no design pass yet; "Glassmorphism" UI polish is deferred to Phase 19 (`NEXT_PHASE.md`).
- **Single GHA workflow file, no tests, no CI lint gate:** `package.json` defines `lint` but there's no GitHub Action running it; there is no test runner dependency (no jest/vitest/playwright). Verification is `npx tsc --noEmit` only.
- **Hard dependency on GitHub API availability:** if `api.github.com` is down or the `GITHUB_PAT` lacks `contents: write`/`actions: write` scope, session triggering fails (`TRANSITION_SNAPSHOT.md` flags this as a risk).

## Impact or Value

For the intended 10–50 Pakistani student alpha cohort, the value proposition (if the extraction/persistence seam is wired) is genuinely high: a personalized, AI-augmented internship agent that costs nothing to run, surfaces roles matched to the student's actual projects and skills rather than keywords, explains *why* each role fits, and learns from Save/Reject feedback to improve future sessions — capabilities normally found only in commercial products. The architecture's emphasis on $0 operations, deterministic-first processing, AI caching, and the creative use of GitHub Actions as a free long-running worker makes it a reproducible blueprint for resource-constrained student-built AI agents. Beyond the immediate user value, the repo is a substantial portfolio/engineering artifact: 17 documented development phases, 11-table relational schema with RLS, a multi-provider search abstraction with circuit breakers and failover, a deterministic-plus-AI hybrid matching engine with a closed feedback loop, webhook-driven idempotent email notifications, and a clean separation between a stateless UI tier and an asynchronous worker tier. `Inferred:` the project's secondary value is pedagogical — it serves as the author's vehicle for learning agentic system design, Supabase, Next.js App Router, GitHub Actions orchestration, and LLM cost engineering in a single end-to-end system. The planned Phase 18 "First Real Run" is explicitly designed to convert this from an architecture-essay-with-code into a validated, live product, and `PHASE_18_ARCHITECTURE.md` §10 lists its success criteria: worker runs start-to-finish without crashing, live internships are scraped/extracted/deduplicated/inserted, the MatchEngine scores them correctly, the webhook fires Nodemailer, and the user sees matches in production — at which point the project's documented ambitions would be realized.
